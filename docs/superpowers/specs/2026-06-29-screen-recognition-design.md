# 화면 인식(Screen Recognition) 설계 — 티파고

작성 2026-06-29. 스파이크(throwaway)로 핵심 가정 검증 후 작성.

## 1. 목표
화면공유로 인게임 티카투카 창을 캡처해 **보드 상태와 내가 굴린 주사위를 자동 인식**, 기존 추천 엔진에 입력한다. 수동 입력을 대체하는 "또 하나의 입력 방식"이며, **솔버/엔진/기존 UI는 변경하지 않는다.**

## 2. 핵심 사실 / 제약 (게임)
- 티카투카는 로스트아크 **내부 미니게임 창**. OS상 별도 창이 아니라, 화면공유에선 **"창 → 로스트아크"**를 골라야 한다. 창 밖 배경은 수시로 바뀐다.
- 창은 **고정 크기 + 드래그 이동**. 위/좌/우로는 화면 밖으로 안 나가고 **아래로만 잘릴 수 있다**. (상단 타이틀바·로고는 항상 보임)
- 해상도/UI 크기 설정에 따라 창의 **픽셀 배율이 달라질 수 있다.**
- 주사위: 값은 점 패턴(2~6은 표준 주사위, **1은 모코코 씨앗 특수 이미지**). 점 색 = 소유(초록=나/빨강=상대). **테두리 = 실드**(초록=내 실드/빨강=상대 실드), 실드는 **양쪽 필드 어디에나** 놓일 수 있다.
- 보드 = 3라인 × (내 3칸 | 합 | 화살표 | 합 | 상대 3칸). 좌=내 필드, 우=상대 필드.
- **좌측 홀딩박스에 주사위가 있으면 = 내 턴**, 그 주사위가 "내가 굴린 주사위". 상대(AI)는 5초 내 빠르게 둠 → 상대 굴린 주사위는 잡을 필요 없음(보드 재인식으로 반영).
- 라인의 **흰 테두리 = 배치 가능**. 알까기 보너스 보유 시 상대 라인에도 흰 테두리가 들어옴 → 보너스 배치 모드 신호.
- 엔진에 필요한 건 칸별 **(필드 위치, 값, 실드 여부)**뿐. 점 색(소유)은 보조 신호로만 사용.

## 3. 아키텍처
인식은 **추가식**이라 실패해도 수동 입력으로 폴백, 본체 영향 0.

새 모듈 `src/vision/` (대부분 순수함수, 브라우저 의존부는 capture 하나로 격리):
- **capture.js** — `getDisplayMedia` 스트림 관리 + 프레임을 캔버스로 추출(ImageData). 유일한 비결정/브라우저 결합부.
- **anchor.js** — 프레임에서 상단 랜드마크(로고/타이틀바)를 **다중배율 매칭**으로 찾아 `{x, y, scale}` 반환. (순수)
- **layout.js** — 앵커 `{x,y,scale}`로 18칸·좌/우 홀딩박스·라인테두리 영역의 픽셀 사각형 계산. 격자는 **앵커 기준 고정 오프셋 상수 × scale**. (순수)
- **templates.js** — 기준 스프라이트(빈칸·1씨앗·2~6·실드 테두리)를 회색조·정규화로 보관. 면당 여러 샘플 평균.
- **recognize.js** — **핵심·순수**: `(frame, layout, templates) → { board:{me/opp 라인[{value,shield}]}, rolledDie, isMyTurn, bonusMode, clipped, confidence[18] }`.
- **scan-controller.js** — 반자동 루프: 주기 프레임 → 변화 감지 → 디바운스 → recognize → 상태변경 이벤트. 무거우면 매칭을 Worker로.

데이터 흐름:
```
스트림 → capture.grabFrame → anchor(위치·배율) → layout(칸 위치)
       → recognize(값·실드·턴·잘림·confidence) → 변화감지·디바운스
       → app 상태 갱신(검토 오버레이) → (확정) → 기존 solve()
```

## 4. 캡처 (capture.js)
- 버튼 "스캔 시작" 클릭(사용자 제스처 필수) → `getDisplayMedia({video})` → 사용자가 **창 → 로스트아크** 선택. 자동 선택 불가(매 세션 1회).
- `<video>`에 스트림 연결, `requestVideoFrameCallback` 또는 타이머로 프레임을 offscreen `<canvas>`에 그려 `getImageData` → recognize 입력.
- "스캔 중지"로 트랙 정지. 스트림 종료 이벤트 처리.

## 5. 앵커링 (anchor.js) — **검증됨**
- 랜드마크 = **창 상단의 로고/타이틀바 패치**(아래로 잘려도 항상 보임).
- **다중배율 매칭**: 랜드마크 템플릿을 여러 배율(예 0.5~1.6, 단계 탐색)로 회색조 SAD 매칭, coarse-to-fine로 최소 오차 위치·배율 선택. 검출된 `scale`로 이후 격자·템플릿을 스케일.
- 스파이크 결과: 창 이동(+350px)·하단 잘림(+451px)·70% 축소(배율 0.7 정확 검출) 모두 통과.
- 매칭 오차(perPixel)가 임계 초과면 "보드를 못 찾음" → 사용자 안내.

## 6. 레이아웃·보정 (layout.js)
- 격자는 **앵커 원점 기준 고정 오프셋(상수표) × scale**. 매 프레임 재검출하지 않음(빈 칸 있으면 검출 실패하므로). 상수표는 기준 해상도(2560×1440, 배율 1)에서 1회 측정.
- 18칸 중심 + 셀 크기 + 좌/우 홀딩박스 + 라인 테두리 영역을 산출.
- **각 칸은 중앙만 사용**(테두리·주사위 사이 연결 아이콘 오염 회피).

## 7. 인식 (recognize.js) — **검증됨(점세기 기각, 템플릿 매칭 채택)**
- 각 칸 패치를 **회색조·제로민 정규화** 후, 템플릿(빈칸·1~6)과 **SSD 최소** → 값. (색 무관하게 패턴으로 매칭, 씨앗=1 정확 구분)
- **confidence margin** = (2등 SSD − 1등 SSD)/1등. margin이 낮은 칸 = 오답 위험 → **검토 표시 트리거**.
- **실드**: 칸 테두리 링의 초록/빨강 채도 비율로 판별. (값과 독립)
- **빈 칸**: 빈칸 템플릿이 최저이거나 중앙 밝기 낮음.
- 정밀화(스파이크에서 식별): ① 격자 중심 정밀화(특히 3번째 줄), ② 면당 템플릿 여러 장 평균, ③ 실드 판별 임계 튜닝.

## 8. 턴 / 굴린 주사위
- 좌측 홀딩박스 인식: 주사위 있으면 `isMyTurn=true`, 그 값이 `rolledDie`. 비었으면 내 턴 아님.
- `bonusMode`: 상대 라인에 흰 테두리가 감지되면 true(알까기 보너스 배치 상태) → 기존 보너스 모드와 연결.

## 9. 잘림 처리 (clipping)
- 앵커는 상단 기준이라 하단 잘림에도 위치를 잡음. layout이 만든 각 칸 사각형이 **프레임 경계를 벗어나면** 그 칸은 "안 보임"으로 표시.
- 보이지 않는 칸이 있으면 `clipped=true` → "보드가 잘렸어요. 창을 위로 올려주세요" 안내(오인식 대신).

## 10. 반자동 루프 (scan-controller.js)
- 주기(예 2~4 fps)로 프레임 취득. **변화 감지**: 칸 영역 해시/차분으로 보드가 바뀌었는지 판단.
- 바뀐 뒤 **디바운스**(예 N프레임 안정 = 애니메이션 종료)되면 recognize 실행 → 상태 변경 이벤트. (주사위 굴림 애니메이션 중 오인식 방지)
- **기본 동작(반자동)**: 안정된 인식이 나오면 **자동으로** 기존 보드(`st`)·`die`에 반영하고 `solve()`까지 실행(별도 확정 클릭 없이). 단, **낮은 confidence 칸이 있으면 자동 반영을 보류**하고 강조 표시 → 사용자가 확인/수정 후 진행. (모든 칸이 충분히 확신될 때만 무인 자동.)

## 11. 검토 / 보정 UX
- 인식된 보드를 기존 보드 UI 위에 표시, **낮은 confidence 칸을 강조**(예 노란 테두리).
- 사용자가 칸을 클릭해 수동 수정 가능(기존 입력 UI 재사용). 자동 입력과 수동 편집 충돌은 "검토 버퍼 → 확정" 단계로 분리.

## 12. 기존 앱 연결
- `app.js`에 스캔 시작/중지 버튼 + 검토 오버레이. 확정 시 `buildEngineState`가 쓰는 기존 구조에 기록 → **솔버/워커는 그대로**.
- 인식이 채우는 값 = 칸 위치(필드)+값+실드. 점 색(소유)은 무시(엔진엔 불필요).

## 13. 에러 / 엣지
- 보드 못 찾음(앵커 오차 큼) → 안내. 잘림 → 안내. 스트림 끊김 → 스캔 중지 상태로.
- 브라우저: 데스크톱 크롬/엣지 + HTTPS(Vercel) 필요. 모바일 미지원 안내.
- 창 캡처는 다른 창에 가려져도 표면을 직접 가져옴(가려짐 면역).

## 14. 테스트 전략
- **recognize.js는 순수함수** → `vision-fixtures/*.png`를 디코드해 결정적 단위테스트(기존 `node --test` 규율과 동일).
- PNG 디코드는 Node 내장 `zlib` 기반 무의존성 디코더(테스트 전용). 브라우저는 canvas 사용.
- 케이스: 빈/중반/보너스/씨앗/실드 양색/창 이동/하단 잘림/축소(해상도). 기대 보드를 픽스처별로 단언.
- anchor/layout도 전체 프레임 픽스처로 테스트.

## 15. 스파이크에서 검증된 것 / 남은 정밀화
검증: ① 무의존성 PNG 디코드 ② top-landmark 앵커(이동·잘림) ③ 다중배율 앵커(해상도) ④ 템플릿 매칭 인식(값 16/18, 씨앗 정확) ⑤ margin=오답 신호.
남은 정밀화(방법 아닌 보정): 격자 중심 정밀화, 면당 템플릿 평균, 실드 판별 튜닝, 변화감지·디바운스 파라미터.

## 16. 범위 밖 / 향후
- 상대 굴린 주사위 실시간 캡처(불필요 — 보드 재인식으로 충분).
- 모바일/다른 브라우저.
- 실제 다른 해상도 스샷으로 최종 검증(현재는 축소본으로 대리 검증).
