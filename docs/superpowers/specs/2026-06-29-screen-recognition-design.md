# 화면 인식(Screen Recognition) 설계 — 티파고

작성 2026-06-29. 스파이크(throwaway)로 핵심 가정 검증 후 작성. **v2: 아키텍트 검토 반영**
(좌우 packed 매핑, 자동적용 정책 단일화, 비인식 입력, 세션당 배율 고정+워커 경계, frame 인터페이스/템플릿 저장 규칙, 검증 주장 정직화).

## 1. 목표
화면공유로 인게임 티카투카 창을 캡처해 **보드 상태와 내가 굴린 주사위를 자동 인식**, 기존 추천 엔진에 입력한다. 수동 입력을 대체하는 "또 하나의 입력 방식"이며, **솔버/엔진/기존 UI는 변경하지 않는다.**

## 2. 핵심 사실 / 제약 (게임)
- 티카투카는 로스트아크 **내부 미니게임 창**. 화면공유에선 **"창 → 로스트아크"** 선택. 창 밖 배경은 수시로 바뀐다.
- 창은 **고정 크기 + 드래그 이동**. 위/좌/우로는 안 나가고 **아래로만 잘림**. 상단 타이틀바·로고는 항상 보임.
- **로아는 테두리없는 창모드(borderless windowed)** 가정 — 전체화면 독점은 창 캡처가 검은 화면/실패가 될 수 있음.
- 해상도/UI 크기 설정에 따라 창 **픽셀 배율**이 달라질 수 있음. 단, **한 세션 중에는 배율 불변**(창은 이동만, 재크기 안 함).
- 주사위: 값=점 패턴(2~6 표준, **1은 모코코 씨앗 특수 이미지**). 점 색=소유(초록 나/빨강 상대) — **엔진엔 불필요**. **테두리=실드**(초록 내 실드/빨강 상대 실드), 실드는 **양쪽 필드 어디에나**. **실드 테두리 색(소유)도 엔진엔 무시** — 실드 여부(boolean)만 필요(rules.js는 `!d.shield`만 봄).
- 보드 = 3라인 × (내 3칸 | 합 | 화살표 | 합 | 상대 3칸). 좌=내 필드, 우=상대 필드.
- **좌측 홀딩박스에 주사위 = 내 턴**, 그 주사위가 "내가 굴린 주사위". 상대(AI)는 빠르게 둠 → 상대 굴린 주사위 캡처 불필요.
- 라인 **흰 테두리 = 배치 가능**. 알까기 보너스 보유 시 상대 라인에도 흰 테두리 → 보너스 배치 모드 신호.

## 3. 아키텍처
인식은 **추가식** — 실패해도 수동 입력 폴백, 본체 영향 0.

새 모듈 `src/vision/` (브라우저 결합부는 capture 하나로 격리; **나머지 순수 모듈은 브라우저 전역 접근 금지**):
- **capture.js** — `getDisplayMedia` 스트림 + 프레임 추출. 유일한 브라우저 결합부.
- **frame 인터페이스(고정)**: `{ data: Uint8ClampedArray(RGBA), width, height }` — Node(zlib 디코더)와 브라우저(ImageData)에서 **동일 형태**. anchor/layout/recognize 전부 이걸 입력.
- **anchor.js** — 상단 랜드마크를 매칭해 `{x,y,scale}`. 다중배율은 §5 정책.
- **layout.js** — `{x,y,scale}`로 18칸·좌/우 홀딩박스·라인테두리 사각형 계산(앵커기준 고정 오프셋 상수 × scale).
- **templates.js** — 기준 스프라이트(빈칸·1씨앗·2~6·실드테두리)를 **커밋된 크롭 패치**로 저장(작은 PNG 또는 JS/JSON 픽셀배열; Node는 zlib, 브라우저는 canvas 디코드). 풀프레임 금지. 면당 여러 샘플 평균.
- **recognize.js** — **핵심·순수**: `(frame, layout, templates) → { cells, rolledDie, isMyTurn, bonusMode, clipped, confidence }`. `cells`는 §7의 **공간 칸 형식**.
- **adapter.js** — recognize 결과를 기존 reactive `st`로 변환(§7 packed 매핑, §9 자동적용 정책).
- **scan-controller.js** — 반자동 루프(§10). 매칭은 **vision 전용 Worker**(솔버 worker.js와 별개)에서. 메인: rVFC→canvas→`getImageData`→`ImageData.data` ArrayBuffer를 **transfer(zero-copy)** 로 vision 워커에 전달.

데이터 흐름:
```
스트림 → capture(frame) → [vision worker] anchor(위치·배율) → layout(칸) → recognize(공간칸·턴·잘림·conf)
       → adapter(packed st 변환) → 변화감지·디바운스 → 자동적용 정책(§9) → 기존 solve()
```

## 4. 캡처 (capture.js)
- "스캔 시작"(사용자 제스처) → `getDisplayMedia({video})` → 사용자가 **창 → 로스트아크** 선택(매 세션 1회, 자동 불가).
- `<video>`에 스트림, `requestVideoFrameCallback`(크롬/엣지 데스크톱)로 offscreen canvas에 그려 `getImageData` → frame.
- **기준 해상도/배율은 "창 표면(window backbuffer)" 기준**(모니터 아님 — 창 캡처는 창 자체 백버퍼를 줌).
- **검은 프레임 감지**(전체평균 거의 0) → "전체화면 독점 모드면 테두리없는 창모드로 바꿔주세요" 안내. 스트림 종료 이벤트 처리.

## 5. 앵커링 (anchor.js) — 이동·잘림 검증됨 / 실해상도 미검증
- 랜드마크 = 창 **상단 로고/타이틀바 패치**(아래로 잘려도 항상 보임). 회색조 SAD, coarse-to-fine.
- **세션당 배율 고정**: 세션 시작/앵커 분실 시에만 **다중배율 전체 탐색**으로 `위치+배율` 1회 확정. 이후 프레임은 **배율 고정**, **상단 밴드에서 위치만** 저비용 재탐색(창 이동 추적). 위치 재탐색 실패 시에만 다중배율 재실행.
- **성능 예산**: 풀프레임 다중배율 = 세션당 1회. 프레임당 = 상단밴드 위치탐색만 → 라이브 루프 경량(§10 fps 충족 목표). 라이브 워커 루프 성능은 §15 빌드 4단계에서 별도 증명.
- 매칭 오차(perPixel) > 임계(초기 ~30) → "보드 못 찾음" 안내.
- 검증: 이동(+350px)·하단잘림(+451px)·**축소본 0.7 배율 정확 검출**. *실제 다른 해상도 렌더링은 미검증(축소 대리)*.

## 6. 레이아웃·보정 (layout.js)
- 격자 = **앵커 원점 기준 고정 오프셋 상수표 × scale**. 매 프레임 재검출 금지(빈 칸 있으면 검출 실패).
- 상수표는 **기준 창 픽셀크기**에서 1회 수동 측정 → 취약 데이터(게임 UI 패치 시 재측정 필요)로 인지.
- 18칸 중심 + 셀 크기 + 좌/우 홀딩박스 + 라인테두리 영역 산출. **각 칸은 중앙만 사용**(테두리·연결아이콘 오염 회피).

## 7. 인식 (recognize.js) + packed 매핑 (adapter.js)
- **칸 값**: 패치를 회색조·제로민 정규화 후 템플릿(빈칸·1~6)과 **SSD 최소** → 값. 색 무관 패턴 매칭(씨앗=1 정확 구분).
- **confidence margin** = (2등 SSD − 1등 SSD)/1등. 임계(초기 ~2) 미만 = 저신뢰 → 검토 트리거. **임계는 미검증 → 다중 픽스처로 보정.**
- **실드**: 칸 테두리 링의 초록/빨강 채도 비율(임계 초기 0.25). 값과 독립. 색(소유) 무시.
- **빈 칸**: 빈칸 템플릿 최저 또는 중앙 밝기 낮음.
- **recognize 출력 = 공간 칸 형식**: 각 라인 각 측면에 **3 공간 칸(좌→우)** `{value:0~6, shield, conf}`. packing은 adapter가.

### 7.1 공간 칸 → packed 라인 매핑 (adapter.js) — **정확성 핵심**
엔진 `st.me[li]`/`st.opp[li]`는 *packed 배열*(빈칸 없음, index 0=먼저 놓인 주사위). 화면은 3 공간 칸. 변환 규칙:
- **채움 방향**(ui-layout.js와 일치): **내 필드 = 우→좌**(첫 주사위=cell2), **상대 = 좌→우**(첫 주사위=cell0).
- packed = 채워진 공간칸을 **채움 순서대로** 나열(내: cell2→1→0 중 채워진 것, 상대: cell0→1→2 중 채워진 것), 각 `{value, shield}`.
- **불가능한 갭 검출**: 채워진 칸이 채움-시작쪽부터 *연속*이 아니면(예 내 필드 [cell0채움, cell1빈, cell2채움]) = 물리적으로 불가 → **오인식**으로 간주, 해당 라인 저신뢰 표시·자동적용 보류.
- 화면칸↔인덱스 매핑은 **ui-layout.js 한 곳**에 유지(재사용/확장; CLAUDE.md 규칙).

## 8. 턴 / 굴린 주사위 — **부분 검증(긍정 케이스만)**
- 좌측 홀딩박스 인식: 주사위 있으면 `isMyTurn=true`, 값=`rolledDie`. 비었으면 내 턴 아님.
- **검증 상태**: 모든 픽스처가 "내 턴"(홀딩박스 채움)이라 긍정 케이스만 가능. **"상대 턴(빈 홀딩박스)" 픽스처 필요**(§15 빌드 게이트).
- `bonusMode`: 상대 라인 흰 테두리 감지 시 true. **소유권은 §9**.

## 9. 비인식 입력 & 자동 적용 정책 (§10/§11 단일화)
- **인식 불가 입력은 수동 유지**: `myMitjang`·`oppMitjang`·`precise`·`realAI`는 보드에 없음 → **수동 토글 그대로**. 자동 계산은 현재 수동값 사용. 특히 **밑장빼기는 종반 exact 답을 바꾸므로**(CLAUDE.md) 사용자 책임으로 명시.
- **자동 적용(단일 흐름)**: scan-controller가 인식 → **검토 버퍼**. 안정(디바운스) + **모든 칸 고신뢰 + 갭 정상 + 미잘림**이면 → `pushHistory()` 후 `st`/`die` 일괄 기록 + `bonusMode` 설정 + `solve()` 자동 실행. 하나라도 저신뢰/갭이상/잘림 → **자동적용 보류**, 오버레이 강조 → 사용자 확인·수정.
- **소유권**: 스캔 활성 중 vision이 `st`/`die`/`bonusMode` 주 기록자. **사용자가 수동 편집하면 그 입력 동안 자동적용 일시중지**(수동 우선). `bonusMode`는 vision이 흰테두리로 설정하되, 스캔 중에는 app의 기존 "배치 후 자동 해제"(app.js setSlotValue)와 충돌하지 않게 vision이 권한 가짐.
- **undo 유지**: 일괄 기록은 반드시 `pushHistory()` 경유.

## 10. 반자동 루프 (scan-controller.js)
- 주기(목표 2~4 fps) 프레임. **변화 감지**: 칸 영역 해시/차분. 바뀐 뒤 **디바운스**(N프레임 안정=애니메이션 종료)되면 recognize.
- 애니메이션(주사위 굴림·알까기 튕김·점수 팝업) 중 오인식 방지. **안정 미수렴 타임아웃**: 일정 시간 안정 안 되면 "인식 보류" 표시.
- 안정 인식 → §9 자동 적용 정책.

## 11. 검토 / 보정 UX
- 인식 보드를 기존 보드 UI 위에 표시, **저신뢰/갭이상/잘림 칸 강조**(노란 테두리).
- 사용자가 칸 클릭해 수동 수정(기존 입력 UI 재사용). §9대로 검토버퍼→자동확정(고신뢰) 또는 수동확정(저신뢰).

## 12. 기존 앱 연결
- `app.js`에 스캔 시작/중지 + 검토 오버레이. 확정 시 reactive `st`(app.js `{me:[[],[],[]],opp}`)에 기록 → 기존 `buildEngineState`가 `{lines,hasMitjang}` 래핑 → **솔버/워커 불변**.
- adapter는 packed `[{value,shield}]`를 `st.me[li]`/`st.opp[li]`에 기록(§7.1), `die.value=rolledDie`, `bonusMode` 설정 — 모두 **pushHistory 경유**.
- `solve()`는 `!die.value`면 조기반환(app.js) → 자동 적용 시 die 먼저 설정.

## 13. 에러 / 엣지
- 보드 못 찾음(앵커 오차>임계) → 안내. 잘림(§9) → "창 위로" 안내. 스트림 끊김 → 스캔 중지. 검은 프레임(§4) → 창모드 안내.
- 브라우저: 데스크톱 크롬/엣지 + HTTPS(Vercel). 모바일 미지원 안내. 창 캡처는 가려짐 면역(단 전체화면 독점 제외, §4).

## 14. 테스트 전략
- **recognize/anchor/layout/adapter는 순수** → `vision-fixtures/*.png`를 무의존성 zlib 디코더로 읽어 **결정적 단위테스트**(`node --test`). 브라우저는 canvas.
- **frame 인터페이스 고정**(§3) — Node/브라우저 동일 형태라야 recognize 이식 가능.
- **템플릿 = 커밋된 크롭 패치**(§3). 풀프레임 픽스처는 repo/배포 비대 → 테스트 입력용으로만, 템플릿 자산은 작게.
- 케이스: 빈/중반/보너스/씨앗/실드양색/이동/하단잘림/축소(해상도) + **음성 경로**: 앵커실패·잘림·저신뢰 게이팅·불가능갭.
- **빌드 게이트(누락 픽스처 수집)**: ① **상대 턴(빈 홀딩박스)** 1장, ② **실제 다른 해상도** 1장. (둘 다 현재 없음.)

## 15. 검증 상태 & 빌드 순서
**검증됨(스파이크)**: 무의존성 PNG 디코드 / top-landmark 앵커(이동·잘림) / 다중배율(축소 *대리* 검증) / 템플릿 매칭(1 픽스처 18칸 중 값 16) / margin이 오답과 상관(n=1).
**미검증/부분**: 턴 감지(긍정만), **실제 다른 해상도**, confidence 게이팅 임계, **라이브 워커 루프 성능**.

**빌드 순서(리스크 우선 재배치)**:
1. **픽스처 잠금** — 상대턴·실해상도 수집(§14 게이트).
2. **데이터 계약 확정** — frame 인터페이스, 공간→packed 매핑(ui-layout 재사용), recognize→st 어댑터(pushHistory/die/bonusMode/mitjang 정책 §9).
3. **순수 모듈**(anchor/layout/recognize/adapter) + 픽스처 테스트 — 스파이크 기반 저위험.
4. **라이브 워커 루프 성능 스파이크**(2~4fps, 배율 고정, ImageData transfer) — **최대 미검증 리스크 먼저 증명.**
5. **UI/오버레이 + 자동적용 흐름**(§9).

**가장 위험한 미지수(순):** ① 라이브 루프 CPU 비용 ② 턴/굴린주사위(미검증+픽스처 없음) ③ 실 해상도/DPI ④ 애니메이션 미수렴 ⑤ 자동적용 제어흐름.

## 16. 범위 밖 / 향후
